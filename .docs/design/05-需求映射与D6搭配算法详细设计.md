> 对应 SSOT D5（需求映射）+ D6（搭配算法）+ P5。build_pc 的完整算法在此钉死，执行期无需再设计。

---

## 1. 需求解析（自然语言 → 结构化需求）

### 1.1 输入

build_pc 收到的是**已由 claude 归一过的结构化参数**（claude 负责语义理解，Python 负责数值）：

```
build_pc(budget_min, budget_max, goal, exclude?)
goal = { game?, resolution?, quality?, fps_target?, usage?, prefer? }
exclude = ["monitor","keyboard",...]   # 不含外设
```

- claude 从 QQ 原话（「2K 中高画质流畅玩生化危机9，预算8000~10000，不要显示器」）解析出：
  `budget_min=8000, budget_max=10000, goal={game:"生化危机9", resolution:"2k", quality:"high", fps_target:60}, exclude=["monitor","peripherals"]`
- Python 不做 NLP，只消费结构化字段。

### 1.2 需求映射查表（D5）

```
1. game 归一（小写+别名表 aliases 匹配）→ 命中 demand_map 行
2. 按 resolution + quality 取该行 min/rec 的 cpu_tier/gpu_tier/vram/ram
3. 命中 → demand_hit = {source:"map", min_gpu_tier, rec_gpu_tier, ...}
4. 未命中游戏 → 回退档位映射（§1.3），demand_hit={source:"fallback", tier_band}
```

### 1.3 回退档位（demand_map 未收录的游戏/场景）

按 resolution×quality 给经验 tier 门槛（GPU/CPU 天梯分档，随天梯刷新校准，初值写入 demand_map 的通用行 game='__generic__'）：

| 场景 | 目标 GPU tier 档 | 目标 CPU game tier 档 | min VRAM | min RAM |
|------|:---:|:---:|:---:|:---:|
| 1080p 中画质 | 中 | 中 | 8 | 16 |
| 1080p 高/2K 中 | 中高 | 中高 | 8~12 | 16 |
| 2K 高画质 | 高 | 高 | 12 | 16~32 |
| 4K 高/极致 | 顶 | 高 | 16 | 32 |

> 档位对应的具体 tier 分数阈值存 demand_map `__generic__` 系列行，P5 建表时随天梯实测填入；执行期查表得数，不临场估。

## 2. D6 搭配算法（硬编基线 + 复核微调）

### 2.1 总体：预算分配基线法

固定「预算比例基线」，按预算切分到各品类，再在每品类预算内选最优件，最后兼容校验 + 预算收敛。

#### 基线比例（游戏向，写死可调，落 env `BUDGET_PROFILE_*`）

| 品类 | 游戏向占比 | 说明 |
|------|:---:|------|
| gpu | 38% | 游戏核心，优先级最高 |
| cpu | 18% | |
| mainboard | 10% | |
| memory | 8% | |
| ssd | 8% | |
| psu | 6% | |
| cooler | 4% | |
| case | 5% | |
| 余量 buffer | 3% | 收敛用 |

> 提供 2 套 profile：`gaming`（上表）、`balanced`（gpu 30/cpu 22...）。goal.usage 决定用哪套，默认 gaming。

### 2.2 选件算法（每品类）

```
budget_cat = total_budget * ratio[cat]
候选 = hardware[category=cat, active=1, effective_price 有效]
      过滤：effective_price ≤ budget_cat * 1.25（允许略超，靠收敛回收）

对 gpu/cpu：满足需求 tier 门槛优先
  1. 先取 tier_score ≥ demand.rec_tier 的最便宜件（够用即止，不盲目堆料）
  2. 若该档超预算，降到 tier_score ≥ demand.min_tier 的最高分件
  3. 仍超 → 记 constraint_violation（预算不足以达标，见 §2.5）

对其他品类：在 budget_cat 内选「热度高 + 规格达标」件
  - memory：容量 ≥ demand.min_ram_gb，类型跟随 CPU/主板
  - ssd：容量 ≥ 500GB 起，NVMe 优先
  - psu：rated_w ≥ est_power*1.5 优先（退而 *1.3），认证/模组作次级排序
  - mainboard：芯片组跟随 CPU（K 系→Z，非 K→B），mem_type 匹配内存
  - cooler：支持 CPU 插槽 + 高度 ≤ 机箱；高 TDP CPU 优先塔散/水冷
  - case：支持主板板型 + 容纳 gpu 长度/散热高度
```

### 2.3 依赖顺序（选件有先后，避免兼容死锁）

```
1. gpu（定性能地基 + 定电源/机箱尺寸需求）
2. cpu（定 socket / mem_type / tdp）
3. mainboard（跟随 cpu socket + mem_type，选芯片组）
4. memory（跟随 mem_type，满足容量）
5. cooler（跟随 cpu socket + tdp）
6. psu（跟随 est_power）
7. case（跟随 mainboard 板型 + gpu 长度 + cooler 高度）
8. ssd/hdd（独立，填满存储预算）
```

### 2.4 兼容修复循环

```
选完跑 validate_build：
  compat_ok=False → 定位 error 规则 → 就近替换冲突件（同预算档换兼容型号）
  最多重试 REPAIR_MAX_ITER 次（默认 5）
  仍不兼容 → 该方案标 compat_ok=False 输出，附 issues（不隐藏失败）
```

### 2.5 预算收敛

```
total = Σ effective_price
if total > budget_max:
    从「性价比边际最低」品类降档（优先 case/cooler/psu 富余处，不动 gpu 达标线）
    重算，最多 CONVERGE_MAX_ITER 次（默认 6）
if total < budget_min:
    向 gpu→cpu→memory→ssd 顺序升档（把预算用足，提升体验），保持 ≤ budget_max
```

### 2.6 多方案输出

固定产 **2~3 个方案**：

| label | 策略 |
|-------|------|
| 「够用」 | 满足 demand.min_tier，偏预算下沿，省钱 |
| 「推荐」 | 满足 demand.rec_tier，预算中位，均衡（默认主推） |
| 「拉满」 | 预算上沿，gpu/cpu 尽量高，体验优先（若 budget 跨度小可省略） |

## 3. claude 复核微调（D6 后半，审计边界内）

Python 产出 2~3 方案后，MCP 把方案连同 issues 返回给 claude；claude 可做**文字性**复核：
- 指出「此价位更推荐 X 而非 Y」的经验判断 → 作为 `perf_note` 附注。
- **不改数字**：如需换件，claude 只能再次调用 build_pc/validate_build（带 prefer 提示），由 Python 重算。claude 直接篡改 total/件价一律禁止（D3 铁律）。

## 6. 模块产物（P5 交付物）

```
app/build/
  __init__.py
  demand.py      # 需求映射查表 + 回退档位

  repair.py      # 兼容修复循环

  build_pc.py    # 编排 + 多方案输出
```

验证（P5 DoD）：
- 「8000~10000 玩生化危机9 2K高画质」→ 产出 2~3 兼容方案、total 落区间、gpu 达推荐档。
- 极窄预算 / 超宽预算 / 未收录游戏（回退档）三类边界均产出合理结果或明确的 constraint_violation。
