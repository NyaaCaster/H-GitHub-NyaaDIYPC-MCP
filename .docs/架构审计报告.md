## 审计-1：数据准确性（价格与规格）

**风险**：best-price-mcp 实测存在三个硬伤（见 `.ref/best-price-mcp-使用说明.md §6`）：
1. 型号串货——查 RTX 5090 返回 5080/5090 混款；
2. 显卡单品与整机/笔记本混淆；
3. 数据源为作者自建腾讯云函数 + 公共 token，无 SLA，可能限流/下线。

**设计缓解**：
- D4 定 **ZOL 京东价为主**（稳定基准，随规格库定时刷新），best-price 仅补淘宝价（次要）。
- 服务端**强制过滤层**：排除整机/笔记本关键词（游戏本/笔记本/主机/整机/台式机）、强制精确型号匹配（区分 5090/5090D/5080）、剔除偏离合理区间的低价。
- best-price 不可用时**自动降级**为纯 ZOL 价，不阻断出方案。

**残留风险**：淘宝补价可能仍有串货漏网。**建议**：淘宝价仅作参考展示，`in_budget` 判定以 ZOL 京东价为准，避免串货低价误导预算判断。

## 审计-2：数字计算可靠性

**风险**：用户明确指出 LLM 在价格计算上有拟合性错误——可能把 1899+1850 算成别的数、或把总价"记成"预算内。

**设计结论 🟢**：D3 铁律正确且彻底——所有加总/预算比对/性价比排序在服务端 Python 完成，`total`/`in_budget`/`compat_ok` 为权威值，claude 只引用不改写。这比"给 LLM 一个计算器工具让它自己调"更稳（后者仍依赖 LLM 正确传参、正确采信返回值）。

**加固建议**：
- 工具返回里**不要**给 claude 留"自己重算"的余地——总价、单价、差额全部预算好，附"请直接引用勿改写"的字段说明。
- 回灌给猫猫的最终话术里，价格数字应来自工具返回原文，可考虑在插件侧对"猫猫复述的总价 ≠ 工具返回总价"做一次断言告警（可选增强）。

## 审计-3：信任边界与鉴权（🔴 高）

**已知高危（记忆 `nyaacli-auth-bypass-finding`）**：NyaaCLI `/task`、`/task/{id}` 端点用 `_=Depends(lambda: _token_guard)`，lambda 返回实例，`__call__` 从不触发，Bearer 校验**从未执行**。叠加 `settings.json` 的 `Bash(*)` 全开 + acceptEdits = 一旦端口暴露即 RCE 入口。当前靠 macmini 私有 LAN 边界遮蔽。

**对本项目的影响**：diy-pc-mcp 挂进 NyaaCLI 的 `.claude/mcp.json` 后，成为该全权限 claude 可调的工具之一。若 diy-pc-mcp 暴露任何写操作（爬虫触发、DB 写、配置改），其安全性**不能依赖 NyaaCLI 的 token 边界**（该边界已失效）。

**审计建议（MUST 纳入 P1/P6）**：
1. **本 MCP 读写分离**：`search_hardware`/`build_pc`/`validate_build` 为纯读工具，不改库；爬虫触发/DB 写走**独立内部端点或 CLI**，不作为 MCP 工具暴露给 claude。
2. 定时爬虫用**容器内 cron/独立进程**触发，不经 MCP 工具面。
3. 若 MCP 走 HTTP 暴露，绑 `127.0.0.1` 或容器内网，不对 LAN/公网开放；如需鉴权自带 `MCP_AUTH_TOKEN` 校验，且**真正校验**（不重蹈 NyaaCLI 的 Depends(lambda) 覆辙）。
4. **旁路修复非本项目职责**，但应在交接文档提示：NyaaCLI 鉴权旁路仍是全局高危，本项目按"边界已失效"假设做防御。

## 审计-4：爬虫合规与反爬

**风险**：ZOL 页面有反爬（无 Referer 重定向到 checking 页）；高频抓取可能被封 IP / 触发法务风险。

**设计缓解**：
- 带 `Referer: https://zj.zol.com.cn/`、合理 UA、请求间隔（限频）。
- 定时刷新（每日 1 次量级），非实时高频轮询。
- GBK → UTF-8 正确解码。

**合规红线**：仅抓公开规格/价格用于**个人装机参考**，不做商业分发、不绕过付费墙、尊重 robots.txt。数据落库仅供猫猫查询，不对外提供 API 转卖。**建议**在 P2 实施前复核 ZOL robots 与使用条款，控制并发 ≤ 1~2、间隔 ≥ 1s。

## 审计-5：搭配算法正确性

**风险**：硬编基线（预算占比 + 逐类选性价比最优）可能产出"通过兼容校验但工程上不合理"的配置，例如：高端 CPU 配丐版主板供电不足、大功耗显卡配额定刚好的电源、ITX 机箱塞进超长显卡（尺寸卡边界）。

**设计缓解**：D6 = 硬编基线 + **claude 复核微调**。claude 拿到基线后可换同价位更合理的件。

**审计判断**：claude 复核是**补丁不是保证**——LLM 可能复核不到位。**建议加固**：
- 兼容规则引擎除硬约束外，加**软约束/告警**：电源冗余系数、供电相数与 CPU 档匹配、显卡长度留 20mm 余量等，基线阶段就规避而非靠 claude 兜。
- 基线算法内置几条"工程常识规则"（如 K 系 CPU 必配 Z/B 系带供电主板），减少对 claude 的依赖。

## 审计-6：需求映射准确性

**风险**：`demand_map` 靠人工维护。**生化危机9 是示例中的新游戏，很可能无实测帧率数据**，档次推定（min_gpu_tier=165 等）可能偏差，导致配高了浪费预算或配低了不达标。

**设计缓解**：映射表每行留 `note` 说明推定依据（同引擎/同代对标）。

**审计建议**：
- 新游戏/无数据时，算法应**明确告知用户"该游戏无实测数据，按同类 3A 大作 2K 需求推定，实际帧率以评测为准"**，不假装精确。
- 提供"性能余量"档（比推定档次高一档）作为保守选项，让用户自行取舍。
- `perf_note` 措辞用"预计"而非"保证"。

## 审计-7：失败模式与降级

需在实现中显式处理并给猫猫**得体话术**：

| 失败 | 降级策略 |
|------|----------|
| best-price 不可用 | 纯 ZOL 价出方案，标注"淘宝价暂缺" |
| 硬件库为空/过期 | 提示"数据刷新中/暂无最新数据"，不出错误堆栈 |
| 预算区间内无兼容解 | 告知"当前预算难以满足 X 需求"，给最接近的方案 + 差多少钱 |
| 需求解析失败（无游戏/分辨率） | claude 追问澄清，或按通用用途出均衡配置 |
| 爬虫某品类抓取失败 | 该品类用上次库存数据，标注 `fetched_at` 时效 |

## 审计-8：成本

**评估 🟢**：
- 猫猫侧 nyaa_think 有并发锁（同用户一次一个）+ 冷却（防刷），已控。
- MCP 查询命中本地 SQLite，开销极小。
- claude 复核微调是一次额外 LLM 调用，可控（单次装机任务量级）。
- 爬虫定时任务离线跑，不占在线请求路径。

**建议**：build_pc 的 claude 复核设为**可选/一次性**，避免多轮反复调 LLM 抬高成本。

---

## 审计-9：数据新鲜度

**风险**：硬件价格波动快（尤其显卡、内存），定时刷新（如每日）之间价格可能已变，报价存在时效性。

**设计缓解**：每条价格带 `priced_at`，`CRAWL_CRON` 可配刷新周期。

**审计建议**：
- 方案返回附**报价时效声明**（如"价格为 X 月 X 日数据，实际以商家为准"）。
- 高波动品类（显卡/内存）可缩短刷新周期或在出方案时对核心件做一次 best-price 实时校价（成本换准确，可配置）。

---

## 审计结论

方案整体**设计合理、决策自洽**，D3 数字计算铁律是最大亮点，直接命中用户核心痛点。

**进入实施前必须落实的三项加固**（纳入对应 P 阶段）：
1. **【P1/P6，高】** 本 MCP 读写分离，写操作不暴露为 claude 可调工具；按"NyaaCLI 鉴权边界已失效"做防御（审计-3）。
2. **【P3，中】** best-price 补价强制过滤层 + 淘宝价仅参考、预算判定以 ZOL 京东价为准（审计-1）。
3. **【P5，中】** 搭配算法内置工程软约束（供电/尺寸余量/K系配Z/B板），不把合理性全押在 claude 复核（审计-5）。

其余中低风险项以"话术得体 + 时效声明 + 留痕可修正"覆盖，随对应阶段实施。
